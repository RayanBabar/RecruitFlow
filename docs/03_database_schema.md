# 3. Database Schema

## 3.1 Overview
The relational database will be managed using Prisma ORM. The following models outline the core entities required for the Smart Job Portal.

```prisma
// This is a conceptual schema to be implemented in Prisma.

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL") // Points to Neon.tech serverless PostgreSQL
}

// ---------------------------------------------------------
// User Roles & Authentication
// ---------------------------------------------------------
enum Role {
  ADMIN
  EMPLOYER
  JOB_SEEKER
}

model User {
  id            String    @id @default(cuid())
  email         String    @unique
  passwordHash  String?
  name          String?
  role          Role      @default(JOB_SEEKER)
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt

  // Relations
  jobsPosted    Job[]         @relation("EmployerJobs")
  applications  Application[] @relation("ApplicantApps")
  profile       Profile?
}

// ---------------------------------------------------------
// Profiles (Job Seekers)
// ---------------------------------------------------------
model Profile {
  id            String   @id @default(cuid())
  userId        String   @unique
  resumeUrl     String?  // Link to PDF storage (e.g., S3 / Vercel Blob)
  parsedData    Json?    // Stores the AI-extracted skills, education, experience
  
  user          User     @relation(fields: [userId], references: [id], onDelete: Cascade)
}

// ---------------------------------------------------------
// Job Postings
// ---------------------------------------------------------
model Job {
  id            String    @id @default(cuid())
  employerId    String
  title         String
  description   String    // Full job description used for AI context
  location      String?
  salary        String?
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt

  employer      User          @relation("EmployerJobs", fields: [employerId], references: [id], onDelete: Cascade)
  applications  Application[]
}

// ---------------------------------------------------------
// Applications & AI Tracking
// ---------------------------------------------------------
enum ApplicationStatus {
  APPLIED
  SHORTLISTED
  REJECTED
}

model Application {
  id            String    @id @default(cuid())
  jobId         String
  applicantId   String
  status        ApplicationStatus @default(APPLIED)
  
  // AI Generated Metadata
  matchScore    Float?    // Generated by FastAPI (0.0 to 100.0)
  aiFeedback    String?   // Justification for the score

  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt

  job           Job       @relation(fields: [jobId], references: [id], onDelete: Cascade)
  applicant     User      @relation("ApplicantApps", fields: [applicantId], references: [id], onDelete: Cascade)
  interviews    InterviewSession[]
}

// ---------------------------------------------------------
// Mock Interview Sessions
// ---------------------------------------------------------
model InterviewSession {
  id            String   @id @default(cuid())
  applicationId String
  startedAt     DateTime @default(now())
  completedAt   DateTime?
  transcript    Json?    // Final record of the LangGraph chat history

  application   Application @relation(fields: [applicationId], references: [id], onDelete: Cascade)
}
```

## 3.2 Key AI Integration Points
* **`Profile.parsedData` (JSON):** This field will store the structured JSON output generated by the FastAPI resume parser. Storing it as JSON prevents brittle relational models for complex, dynamic nested data like lists of technical skills or dynamic job histories.
* **`Application.matchScore`:** This allows Employers to instantly `ORDER BY matchScore DESC` in their applicant dashboard without running AI inference on the fly.
* **`InterviewSession`:** Records the ID that will be passed to the FastAPI WebSocket (`/ws/interview/{session_id}`). When the session concludes, the transcript can be written back via REST or webhook.
